# Nemesis 逐步引導模式設計規格

**日期**：2026-06-14  
**狀態**：草稿  
**範圍**：將現有靜態文件閱讀器改為互動式逐步引導系統，適用於所有遊戲階段

---

## 背景與目標

現有應用以 Markdown 頁面呈現遊戲規則，文字量多且缺乏互動性，新玩家難以跟著執行。新設計目標：

- 每次只顯示一個步驟，降低認知負擔
- 在關鍵步驟提示可能發生的狀況，讓玩家選擇後進入對應處理流程
- 收集遊戲局部狀態（如玩家人數），讓應用判斷階段是否完成
- 全螢幕專注模式，消除導航干擾

---

## 流程架構

### 模式切換

首頁列出所有遊戲階段入口（遊戲設置、玩家回合、事件階段、入侵者階段等）。點擊任一階段後，進入**全螢幕引導模式**：

- Sidebar 完全隱藏
- 整個畫面只顯示當前步驟卡片
- 右上角保留「離開引導」按鈕（點擊後跳出確認提示再返回首頁）

### 步驟推進

每個階段由一個有序步驟陣列組成，使用者操作：

- **「下一步」按鈕**（主要操作，右下）
- **「上一步」文字連結**（次要操作，左下，允許修正）
- **「離開引導」圖示**（右上角，隨時可用）

最後一步完成後顯示結束畫面：「本階段完成 ✓」+ 返回首頁按鈕。

### 狀況處理（中斷層機制）

當步驟類型為 `check` 時，步驟卡片底部列出該步驟可能發生的狀況選項。玩家點擊任一狀況後：

1. 整個畫面被**狀況處理覆蓋層（full-screen overlay）**取代
2. 覆蓋層內有自己的逐步流程（與主流程相同的卡片結構）
3. 狀況流程結束後顯示「✓ 狀況處理完畢，回到主流程」
4. 關閉覆蓋層，**回到觸發狀況的同一個主步驟**，玩家再按「下一步」繼續

覆蓋層內的「上一步」只在狀況步驟間移動，不會返回主流程。「離開引導」按鈕在覆蓋層中同樣可用，點擊後關閉覆蓋層並退出整個引導模式。

玩家可在同一步驟觸發多個狀況（依序處理）。

## Session Context（局部狀態）

引導過程中以 Vue `reactive` 物件維護 session context，整個引導共用：

```js
context = {
  player_count: null,    // 由設置階段 collect 步驟填入
  current_player: 1,     // 玩家回合追蹤用
}
```

**玩家人數收集**：設置階段含一個 `collect` 步驟，呈現「這次遊玩幾位玩家？」選項（2、3、4、5），玩家選擇後存入 `context.player_count`。

**玩家回合追蹤**：玩家回合階段頂部顯示「玩家回合：第 N 位 / 共 M 位」，
每位玩家完成後 `current_player` 遞增，到達 `player_count` 後顯示「所有玩家回合結束」並進入下一步。

未來可依需求擴充 context 欄位（例如：逃生艙是否啟動、特定玩家存活狀態）。

## 步驟卡片 UI

### 視覺結構

```text
┌──────────────────────────────────────────┐
│  [X 離開]               第 3 步，共 8 步  │
│──────────────────────────────────────────│
│                                          │
│  步驟標題                                 │
│                                          │
│  說明文字（1–3 句，簡潔指令語氣）            │
│                                          │
│  ┌─── 這個步驟可能發生以下狀況 ───┐         │
│  │  [遭遇異形]  [發現火警]        │  ← check 型才有
│  └───────────────────────────────┘         │
│                                          │
│  [← 上一步]                [下一步 →]    │
└──────────────────────────────────────────┘
```

- 背景維持現有深色主題（`#111827`）
- 主要操作按鈕使用紅色 accent（`#e63946`）
- 步驟標題使用較大字級，說明文字使用正常字級，字數以 30–60 字為宜
- 玩家回合階段在標題列額外顯示玩家進度（`玩家回合：第 2 位 / 共 4 位`）

### Collect 步驟（玩家人數）

```text
┌──────────────────────────────────────────┐
│  [X 離開]               第 2 步，共 6 步  │
│──────────────────────────────────────────│
│                                          │
│  這次遊玩幾位玩家？                         │
│                                          │
│  [  2  ]  [  3  ]  [  4  ]  [  5  ]    │
│                                          │
│  （選擇後自動進入下一步）                    │
└──────────────────────────────────────────┘
```

## 內容資料模型

現有 Markdown 格式改為 **YAML frontmatter + 步驟陣列**，每個步驟有明確類型。

### 步驟類型

| 類型 | 說明 |
| --- | --- |
| `action` | 純指令步驟，說明要做什麼，按「下一步」繼續 |
| `collect` | 詢問玩家輸入，儲存至 session context，選擇後自動前進 |
| `check` | 步驟底部列出可能發生的狀況按鈕，玩家選擇後觸發覆蓋層 |

### 階段檔案範例

```yaml
---
title: 遊戲設置
order: 1
---

steps:
  - id: setup-1
    type: action
    title: 準備主板
    body: 將主板放置於桌面中央，連接走廊磁磚，確認所有房間標記朝上。

  - id: setup-players
    type: collect
    title: 確認玩家人數
    prompt: 這次遊玩幾位玩家？
    context_key: player_count
    options: [2, 3, 4, 5]

  - id: setup-3
    type: check
    title: 分配角色卡
    body: 每位玩家抽取一張角色卡，取得對應的起始手牌與角色指示物。
    situations:
      - id: malfunction
```

### 狀況檔案格式

狀況（`content/situations/`）同樣改為步驟陣列格式，結構與階段相同，但無 `collect` 類型，且最後一步結束後顯示「回到主流程」而非完成畫面。

## 技術實作方向

### 新增元件

| 元件 | 職責 |
| --- | --- |
| `GuideView.vue` | 全螢幕引導模式主容器，管理步驟索引與 overlay 狀態 |
| `StepCard.vue` | 渲染單一步驟（action / collect / check 三種型態） |
| `SituationOverlay.vue` | 狀況處理覆蓋層，內嵌自己的步驟流程 |
| `useGuideSession.js` | Composable：session context、步驟索引、玩家回合計數 |

### 路由調整

```text
/              → 首頁（階段入口選擇）
/guide/:phase  → 全螢幕引導模式（新增）
```

舊路由 `/phase/:id` 重定向至 `/guide/:id`。

### 現有元件處理

- `PhaseView.vue`：職責移至 `GuideView.vue`，可刪除或留作重定向殼
- `SituationModal.vue`：邏輯合併進 `SituationOverlay.vue`
- `AppSidebar.vue`：引導模式中隱藏（由 `GuideView` 控制父層 class）
- `style.css`：深色主題、紅色 accent 完全保留，新增卡片與 overlay 樣式

### 不需要改動

- `vite.config.js` 設定結構
- `index.html`、字型載入
- 整體 CSS 設計語言
